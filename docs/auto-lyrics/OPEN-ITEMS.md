## 1. CANON-3 — Текстовая нормализация (wanna → want to, gonna → going to)

**Статус:** 🟡 Спроектирован, не внедрён

**Суть:** Genius и LRC-тексты одной песни могут различаться в
сокращениях: "wanna" vs "want to", "gonna" vs "going to". Это
снижает Jaccard similarity в Pass 1, потенциально ухудшая
качество кандидатов.

**Решение (спроектировано):**
```typescript
// Нормализация перед bag-of-words:
// "wanna" → "want to", "gonna" → "going to", "gotta" → "got to"
// Применяется и к Genius-тексту, и к LRC-тексту (симметрично)
```

**Почему отложено:**
- MJ ("They Don't Care About Us") решился Variant B (перебор LRC-версий)
- Runaway не имеет этой проблемы (Genius и LRC используют одинаковые формы)
- Ни один реальный трек не показал бага, вызванного именно этой нормализацией
- Внедрение без реального кейса = риск внести регрессию без пользы

**Когда пересмотреть:**
- Появился трек, где Variant B не помогает, а wanna/gonna разница есть
- Замечено снижение Jaccard для конкретного трека с проверкой "из-за wanna/gonna"

## 2. `_originalTagInferred` — UI-индикация восстановленного тега

**Статус:** 📋 Требует UI-дизайна

**Суть:** При миграции старых PersistedTextBlock (без `originalTag`)
`block-migration.ts` восстанавливает тег по типу блока:
```typescript
result.originalTag = type.charAt(0).toUpperCase() + type.slice(1);
// chorus → "Chorus"
result._originalTagInferred = true; // помечено: не оригинал
```

**Где используется сейчас:** `block-migration.ts` (проставляет),
`gate-b-persistence.smoke.test.ts` (проверяет).

**Где НЕ используется:** UI. Поле `_originalTagInferred` существует
в данных, но нигде в интерфейсе не отображается визуальное отличие
"восстановленный тег" vs "подлинный тег".

**Что нужно:**
- Визуальный индикатор в редакторе блоков (например, italic / тусклый
  цвет / иконка "восстановлено") для тегов, где `_originalTagInferred === true`
- Всплывающая подсказка: "Тег восстановлен автоматически. Оригинал не сохранён."
- Возможно: кнопка "подтвердить тег" для снятия inferred-статуса

**Почему отложено:**
- Функционально работает: блоки маппятся, теги отображаются корректно
- Отсутствие индикации — UX-недостаток, не баг
- Требует дизайн-решения (как именно показать), а не кода

## 3. A3 — Repeated Hook Ambiguity (архитектурно нерешаемо)

**Статус:** 🟡 Признано архитектурно нерешаемым на текущем движке

**Суть:** Если припев и куплет имеют идентичные текстовые строки,
текущий G/L ratio (индексный, не time-based) не может различить,
к какому блоку относится строка.

**Почему нерешаемо:**
```
G/L ratio = 0.6 × textRatio + 0.4 × rankRatio
// rankRatio — индексный (позиция блока в массиве)
// Для time-based нужно: (timeStamp - blockStartTime) / blockDuration
// Но у нас нет time-based привязки блоков — весь pipeline spatial
```

**Когда пересмотреть:**
- Если pipeline получит time-based слой (временные метки блоков)
- Если появится возможность получать duration блоков из внешнего API
- Пока — documented limitation, не баг

## 4. A8 — Long Gap Drift (архитектурно нерешаемо)

**Статус:** 🟡 Признано архитектурно нерешаемым на текущем движке

**Суть:** При большом разрыве между блоками (40+ LRC-строк)
contentMatch в Pass 3 теряет фокус — orphan может привязаться
к semantically неверному блоку.

**Почему нерешаемо:**
- Pass 3 оперирует на уровне отдельных LRC-строк (line-level)
- Для gap > 40 строк блоковое расстояние и line-level расстояние
  перестают коррелировать
- Решение требует block-level candidate retrieval — фундаментальное
  изменение архитектуры Pass 3

**Когда пересмотреть:**
- Появится реальный трек с gap > 40 (сейчас все тестовые треки ≤ 20)
- Или при редизайне Pass 3 на block-level retrieval

## 5. CGP confidence skip threshold (0.9) — не калиброван

**Статус:** 📋 Отложено — работает, не требует срочной калибровки

**Суть:** CGP (Pass 2.5) пропускает блоки с rawScore < 0.9.
Порог установлен эмпирически: на MJ Verse 2 имеет score=0.75,
Bridge 2 NOT MAPPED — CGP не трогает их, и это правильно.

**Риски:**
- Если появится трек, где корректный блок имеет score 0.85
  (недостаточно для CGP), а некорректный — > 0.9 — CGP не поможет
- Если появится трек, где блок с score 0.92 ошибочно корректен —
  CGP будет безуспешно искать замену

**Когда калибровать:**
- При появлении трека с пограничными score (0.85-0.95), где CGP
  ошибается
- Для калибровки: grid search по порогу 0.7-0.95 с шагом 0.05

## 8. Pass 2b borrow cap (50%) — эмпирический, не калиброван

**Статус:** 📋 Отложено — работает на всех известных треках

**Суть:** Pass 2b (positional gap fill for NOT MAPPED) может отнять
строки у следующего MAPPED блока, но не более 50% его ожидаемых
строк. Установлено эмпирически: "меньше половины — блок не сломается".

**Когда калибровать:**
- При появлении трека, где 50% cap слишком жёсткий (gap остаётся
  unfilled при большом next-блоке)
- Или слишком щедрый (next-блок теряет критическую массу строк)
