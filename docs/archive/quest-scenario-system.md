## 1. Purpose

### 1.1 Why Current "Practice Modes" Are Really Scenario Families

The current exercise system (Echo Drill, 3-Take Challenge, Call & Response) reveals a deeper pattern: these are not isolated "practice modes" but **scenario families** — structured variations of a common orchestration grammar.

**Evidence from current implementation:**
- Echo Drill = listen → record pattern with fixed backing
- 3-Take Challenge = record-until-filled pattern with slot targeting
- Call & Response = alternation pattern with in-flight capture

Each "recipe" is actually a **scenario program** generated from a family template with specific parameters (scope, backing, tempo, alternation).

**The problem:** Hardcoded recipes don't scale. Every new practice variation requires:
- New recipe definition
- New step sequence logic
- New runtime orchestration
- New UI surface
- New documentation

This leads to **recipe explosion** — dozens of hardcoded variations that are 80% identical.

### 1.2 Why beLive Needs a Scenario-Generation Architecture

beLive's core value is **adaptive practice** — helping learners master songs through structured repetition with variation. This requires:

1. **Generative scenarios** — not hardcoded recipes, but generated programs from semantic specs
2. **Capability gating** — scenarios adapt to available features (word-sync, pitch, backing stems)
3. **Teacher authoring** — teachers compose scenarios from grammar, not code
4. **Progression models** — scenarios scale difficulty (tempo, backing, scope)
5. **Evidence collection** — scenarios capture mastery signals (X/Y, pitch, timing)

**The shift:** From "5 hardcoded recipes" to "scenario generator families that produce infinite variations from semantic specs."

## 2. Core System Shape

The Quest & Scenario system follows a **four-layer architecture**:

```
Semantic Grammar → Generator Families → Scenario Programs → Quest Wrappers
```

### 2.1 Layer Breakdown

**Layer 1: Semantic Grammar**
- Declarative scenario specification language
- Dimensions: scope, tempo, backing, listen mode, record mode, alternation, review, progress gate, evidence
- Example: `{ scope: "block:chorus", tempo: "0.85x", backing: "instrumental", pattern: "echo", rounds: 3 }`

**Layer 2: Generator Families**
- Rare, meaningful families (not one-per-recipe)
- Each family knows how to generate scenario programs from semantic specs
- Examples: Echo Family, Alternation Family, Until-Filled Family, Tempo-Ladder Family

**Layer 3: Scenario Programs**
- Executable step sequences with runtime contracts
- Generated by families from semantic specs
- Contains: steps, repeat logic, progress gates, evidence requests, interruption policy

**Layer 4: Quest Wrappers**
- Motivational/UI layer around scenario programs
- Provides: name, icon, description, goal framing, progress visualization
- Shareable unit for teachers/learners

### 2.2 Data Flow

```
Teacher/System → Semantic Spec → Generator Family → Scenario Program → Quest Runtime → Evidence
```

1. Teacher composes semantic spec (or system generates from progression model)
2. Generator family validates spec and generates scenario program
3. Quest runtime executes program against Takes substrate
4. Evidence collector captures mastery signals
5. Progression model adapts next scenario based on evidence

## 3. Entity Map

### 3.1 ScenarioSpec

**Purpose:** Declarative specification of a practice scenario  
**Lifecycle:** Authored by teacher or generated by progression model  
**Format:** JSON-serializable object with semantic dimensions

**Schema:**
```typescript
interface ScenarioSpec {
  id: string;
  family: 'echo' | 'alternation' | 'until-filled' | 'tempo-ladder' | ...;
  scope: ScopeSpec;           // block, line-range, full-song
  tempo: TempoSpec;           // 1.0, 0.85, ladder
  backing: BackingSpec;       // full, instrumental, vocals-only, none
  listenMode?: ListenMode;    // reference, guide, none
  recordMode?: RecordMode;    // standard, in-flight, continuous
  alternation?: AlternationPattern;  // call-response, guide-user, etc.
  reviewStep?: ReviewStep;    // compare, self-assess, none
  progressGate?: ProgressGate; // rounds, filled-slots, mastery-threshold
  evidence?: EvidenceRequest[]; // X/Y, pitch, timing, completion
  holdBehavior?: HoldBehavior; // pause-on-error, continue, retry
  scenePolicy?: ScenePolicy;   // visibility, highlight, cue
}
```

**Validation:** Generator family validates spec before program generation

### 3.2 ScenarioGenerator

**Purpose:** Family-specific logic for generating scenario programs from specs  
**Lifecycle:** Singleton per family, registered at app init  
**Responsibility:** Validate spec, generate program, handle capability gating

**Interface:**
```typescript
interface ScenarioGenerator {
  family: string;
  validate(spec: ScenarioSpec): ValidationResult;
  generate(spec: ScenarioSpec, context: GenerationContext): ScenarioProgram;
  capabilities: CapabilityRequirement[];
}
```

**Examples:**
- `EchoGenerator` — generates listen → record programs
- `AlternationGenerator` — generates call-response programs with in-flight capture
- `UntilFilledGenerator` — generates record-until-filled programs with slot targeting


### 3.3 ScenarioProgram

**Purpose:** Executable step sequence with runtime contracts  
**Lifecycle:** Generated by family, executed by quest runtime, disposed after completion  
**Format:** Step array + metadata + contracts

**Schema:**
```typescript
interface ScenarioProgram {
  id: string;
  specId: string;
  family: string;
  steps: ScenarioStep[];
  repeat: RepeatLogic;
  progressGate: ProgressGate;
  evidenceRequests: EvidenceRequest[];
  interruptionPolicy: InterruptionPolicy;
  resumeSemantics?: ResumeSemantics;
}

interface ScenarioStep {
  action: 'listen' | 'record' | 'wait' | 'compare' | 'review';
  scope?: ScopeSpec;
  backing?: BackingSpec;
  slot?: SlotTarget;
  captureMode?: 'standard' | 'in-flight';
  instruction: string;
  duration?: number;
}
```

**Contracts:**
- Interruption policy (pause, cancel, resume)
- Evidence collection points
- Progress gate evaluation
- Hold/fix behavior on error

### 3.4 QuestCard

**Purpose:** Motivational wrapper + shareable unit  
**Lifecycle:** Created by teacher or system, stored in catalog, instantiated as QuestRun  
**Format:** Metadata + ScenarioSpec reference

**Schema:**
```typescript
interface QuestCard {
  id: string;
  name: string;
  icon: string;
  description: string;
  category: 'drill' | 'challenge' | 'exploration';
  difficulty: 'beginner' | 'intermediate' | 'advanced';
  scenarioSpec: ScenarioSpec;
  goalFraming: string;
  estimatedDuration: number;
  requiredCapabilities: CapabilityRequirement[];
}
```

**Sharing:** QuestCard is the shareable unit (not runtime state, not program)


### 3.5 QuestRun

**Purpose:** Runtime instance of a quest execution  
**Lifecycle:** Created on quest start, updated during execution, finalized on completion  
**Format:** Program + state + evidence

**Schema:**
```typescript
interface QuestRun {
  id: string;
  questCardId: string;
  program: ScenarioProgram;
  state: QuestState;
  evidence: Evidence[];
  startedAt: number;
  completedAt?: number;
  status: 'active' | 'paused' | 'completed' | 'abandoned';
}

interface QuestState {
  currentRound: number;
  currentStepIndex: number;
  phase: 'idle' | 'listening' | 'pre-recording' | 'recording' | 'comparing' | 'waiting';
  resolvedTimeRange: TimeRange | null;
  recordSlot: number | null;
}
```

**Separation:** QuestRun is NOT QuestCard, QuestRun is NOT ScenarioProgram

### 3.6 Evidence

**Purpose:** Captured mastery signals from quest execution  
**Lifecycle:** Collected during quest run, stored after completion, used for progression  
**Format:** Type-specific evidence objects

**Schema:**
```typescript
interface Evidence {
  id: string;
  questRunId: string;
  type: 'completion' | 'x-y' | 'pitch' | 'timing' | 'self-assessment';
  timestamp: number;
  data: EvidenceData;
}

type EvidenceData = 
  | CompletionEvidence    // rounds completed, slots filled
  | XYEvidence            // X/Y capture data
  | PitchEvidence         // pitch accuracy, stability
  | TimingEvidence        // rhythm accuracy, sync
  | SelfAssessmentEvidence; // learner rating
```

**Usage:** Evidence feeds progression model, mastery calculation, adaptive difficulty

### 3.7 Mastery

**Purpose:** Aggregated mastery state for a song/block  
**Lifecycle:** Updated after each quest run, persisted across sessions  
**Format:** Mastery scores + progression state

**Schema:**
```typescript
interface Mastery {
  songId: string;
  blockId?: string;
  scores: {
    completion: number;    // 0-1
    accuracy: number;      // 0-1 (pitch, timing)
    consistency: number;   // 0-1 (variance across takes)
    confidence: number;    // 0-1 (self-assessment)
  };
  progressionState: {
    currentTempo: number;
    currentBacking: BackingMode;
    unlockedCapabilities: string[];
  };
  lastPracticed: number;
  totalPracticeTime: number;
}
```

**Usage:** Drives adaptive scenario generation, unlocks new quest cards, shows progress

## 4. Strict Separation Rules

### 4.1 Card is NOT Program

**QuestCard** = shareable metadata + spec reference  
**ScenarioProgram** = executable step sequence

**Why separate:**
- Card can be shared without exposing runtime details
- Same card can generate different programs based on context (capability gating, progression)
- Program is disposable, card is persistent

**Anti-pattern:** Storing program in card, sharing runtime state

### 4.2 Program is NOT Run

**ScenarioProgram** = stateless step sequence + contracts  
**QuestRun** = stateful execution instance

**Why separate:**
- Program can be reused across multiple runs
- Run captures evidence, program defines structure
- Program is generated, run is instantiated

**Anti-pattern:** Mutating program during execution, storing state in program

### 4.3 Quest is NOT Generator

**QuestCard** = motivational wrapper  
**ScenarioGenerator** = family-specific generation logic

**Why separate:**
- Generator is code, card is data
- Generator is singleton, card is many
- Generator validates and generates, card describes and motivates

**Anti-pattern:** Embedding generation logic in card, hardcoding programs in generator

### 4.4 Spec is NOT Program

**ScenarioSpec** = declarative semantic description  
**ScenarioProgram** = imperative step sequence

**Why separate:**
- Spec is human-authorable, program is machine-generated
- Spec is validated, program is executed
- Spec is portable, program is runtime-specific

**Anti-pattern:** Mixing semantic and imperative concerns, skipping validation

## 5. Relation to Takes

### 5.1 Takes = Substrate

The Takes system provides the **execution substrate** for scenario programs:

**Takes provides:**
- Block-scoped recording sessions
- Take slot management (0-5 slots per block)
- Audio capture + trim + decode pipeline
- Take preview/compare infrastructure
- Waveform visualization (live trail, compare overlays)

**Takes does NOT provide:**
- Practice orchestration (that's Scenario)
- Motivational framing (that's Quest)
- Evidence collection (that's Quest/Evidence)
- Progression logic (that's Mastery)

### 5.2 Scenario = Orchestration Core

The Scenario system orchestrates Takes operations:

**Scenario orchestrates:**
- When to start recording (pre-roll, countdown, in-flight)
- Which slot to target (next empty, specific slot, round-locked)
- When to stop recording (block end, intermediate window, manual)
- When to trigger compare (after record, on demand, never)
- What backing to apply (full, instrumental, vocals-only, none)

**Scenario does NOT:**
- Manage take storage (that's Takes)
- Render waveforms (that's Takes)
- Handle audio engine (that's Takes via window.audioEngine)

### 5.3 Quest = Motivational Wrapper

The Quest system wraps scenarios with motivation:

**Quest provides:**
- Name, icon, description, goal framing
- Progress visualization (rounds, steps, completion)
- Evidence collection triggers
- Interruption handling (Escape key, pause, resume)
- Completion celebration

**Quest does NOT:**
- Execute steps (that's Scenario runtime)
- Manage takes (that's Takes)
- Generate programs (that's Generator)

### 5.4 Layering Diagram

```
┌─────────────────────────────────────┐
│  Quest Layer (Motivation + UI)      │
│  - QuestCard, QuestRun, Evidence    │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Scenario Layer (Orchestration)     │
│  - ScenarioSpec, Generator, Program │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│  Takes Layer (Execution Substrate)  │
│  - Recording, Slots, Preview, Mix   │
└─────────────────────────────────────┘
```

**Dependency rule:** Quest → Scenario → Takes (never reverse)

## 6. Scenario Dimensions

Scenario specs are composed from **semantic dimensions** — orthogonal parameters that define practice structure.

### 6.1 Scope

**Purpose:** Define what portion of the song to practice  
**Values:**
- `block:<blockId>` — single block (verse, chorus, etc.)
- `line-range:<start>-<end>` — specific line range
- `full-song` — entire song
- `custom:<markerRange>` — custom time range

**Example:** `{ scope: "block:chorus-1" }`

### 6.2 Tempo Stage

**Purpose:** Control playback speed for difficulty scaling  
**Values:**
- `1.0` — full speed
- `0.85` — 85% speed (common practice tempo)
- `0.70` — 70% speed (slow practice)
- `ladder:[0.7, 0.85, 1.0]` — progressive tempo increase

**Example:** `{ tempo: 0.85 }`

### 6.3 Backing Mode

**Purpose:** Control which audio stems are active  
**Values:**
- `full` — instrumental + vocals (reference)
- `instrumental` — instrumental only (standard practice)
- `vocals-only` — vocals only (pitch guide)
- `none` — silence (a cappella)

**Example:** `{ backing: "instrumental" }`

### 6.4 Listen Mode

**Purpose:** Define how learner hears reference before recording  
**Values:**
- `reference` — full backing with vocals (standard echo)
- `guide` — vocals-only guide (pitch focus)
- `none` — no listen step (direct to record)

**Example:** `{ listenMode: "reference" }`

### 6.5 Record Mode

**Purpose:** Define recording capture strategy  
**Values:**
- `standard` — pre-roll → countdown → record → stop
- `in-flight` — continuous transport, recorder arms at target time
- `continuous` — no stops, multiple windows in single session

**Example:** `{ recordMode: "in-flight" }`


### 6.6 Alternation Pattern

**Purpose:** Define guide/user alternation structure  
**Values:**
- `none` — no alternation (standard practice)
- `call-response` — guide sings line, user sings next line
- `phrase-echo` — guide sings phrase, user echoes same phrase
- `verse-chorus` — guide sings verse, user sings chorus

**Example:** `{ alternation: "call-response" }`

**Note:** Alternation is a **special family** — requires dedicated generator, not a dimension of other families.

### 6.7 Review Step

**Purpose:** Define post-recording review/compare behavior  
**Values:**
- `compare` — automatic compare playback after record
- `self-assess` — prompt learner to rate their take
- `none` — no review, advance immediately

**Example:** `{ reviewStep: "compare" }`

### 6.8 Progress Gate

**Purpose:** Define completion criteria  
**Values:**
- `rounds:<count>` — complete N rounds
- `filled-slots:<count>` — fill N take slots
- `mastery-threshold:<score>` — achieve mastery score
- `time-limit:<seconds>` — practice for N seconds

**Example:** `{ progressGate: "rounds:3" }`

### 6.9 Hold/Fix Behavior

**Purpose:** Define error handling strategy  
**Values:**
- `pause-on-error` — pause transport, show error, wait for user
- `continue` — log error, continue execution
- `retry` — retry failed step automatically

**Example:** `{ holdBehavior: "continue" }`

### 6.10 Evidence Layer

**Purpose:** Define what mastery signals to collect  
**Values:**
- `completion` — rounds completed, slots filled
- `x-y` — X/Y capture data (requires X/Y capability)
- `pitch` — pitch accuracy, stability (requires pitch capability)
- `timing` — rhythm accuracy, sync (requires word-sync capability)
- `self-assessment` — learner rating

**Example:** `{ evidence: ["completion", "x-y"] }`

### 6.11 Scene/Visibility Policy

**Purpose:** Define visual presentation during execution  
**Values:**
- `standard` — default rehearsal scene (block view, word highlight)
- `minimal` — hide lyrics, focus on audio
- `guided` — highlight target lines, dim others
- `quest-overlay` — show quest-specific UI elements

**Example:** `{ scenePolicy: "guided" }`

## 7. Generator Family Principle

### 7.1 Families Are Rare and Meaningful

**Core principle:** Generator families are NOT one-per-recipe. Families represent **fundamental orchestration patterns** that can generate many variations.

**Anti-pattern:** Creating a new family for every slight variation (e.g., "Echo Drill 3 Rounds", "Echo Drill 5 Rounds", "Echo Drill Slow Tempo")

**Correct pattern:** One Echo Family generates all echo variations from semantic specs.

### 7.2 What Makes a Family

A generator family is justified when it has:

1. **Distinct orchestration logic** — fundamentally different step sequencing
2. **Unique runtime contracts** — different interruption, resume, or evidence semantics
3. **Non-composable from other families** — cannot be expressed as parameter variation

**Examples of valid families:**
- **Echo Family** — listen → record pattern (standard practice)
- **Alternation Family** — guide/user alternation with in-flight capture (special orchestration)
- **Until-Filled Family** — record until N slots filled (different progress gate)
- **Tempo Ladder Family** — progressive tempo increase across rounds (experimental)

**Examples of NOT families:**
- "Echo 3 Rounds" vs "Echo 5 Rounds" — same family, different `progressGate` parameter
- "Echo Slow" vs "Echo Fast" — same family, different `tempo` parameter
- "Echo Instrumental" vs "Echo A Cappella" — same family, different `backing` parameter

### 7.3 Initial Candidate Family Set

**Status:** Candidate set, not final fixed count

**Likely families:**
1. **Echo Family** — listen → record pattern (stable, proven)
2. **Until-Filled Family** — record until slots filled (stable, proven)
3. **Alternation Family** — call-response, phrase-echo patterns (special, experimental)
4. **Tempo Ladder Family** — progressive tempo increase (experimental)
5. **Exploration Family** — free-form practice with optional guidance (future)

**Evaluation criteria:**
- Does it have distinct orchestration logic?
- Does it require unique runtime contracts?
- Can it generate meaningful variations from semantic specs?
- Is it composable from existing families?

**Expected outcome:** 3-7 families, not 20+

## 8. Frozen Decisions

These decisions are **accepted and frozen** — implementation must follow these constraints.

### 8.1 Generators as Scaling Unit

**Decision:** Generator families are the scaling unit, not hardcoded recipes.

**Rationale:**
- Recipes don't scale (leads to explosion)
- Generators produce infinite variations from semantic specs
- Teachers author specs, not code

**Implication:** New practice variations are created by authoring specs, not writing code.

### 8.2 Capability Gating First-Class

**Decision:** Scenario generation must respect capability availability.

**Rationale:**
- Not all songs have word-sync, pitch data, or backing stems
- Scenarios must adapt to available features
- Graceful degradation is required

**Implication:** Generators check capabilities before generating programs, fallback to simpler variants if needed.

### 8.3 Interruption Policy Belongs to Program Contract

**Decision:** Interruption behavior (pause, cancel, resume) is part of scenario program contract.

**Rationale:**
- Different scenarios have different interruption semantics
- Alternation scenarios cannot pause mid-window
- Standard scenarios can pause anytime

**Implication:** Program defines interruption policy, runtime enforces it.

### 8.4 Compare Separate from PanoramaMix Lane

**Decision:** Take compare is a separate concern from PanoramaMix (future multi-take audition).

**Rationale:**
- Compare = A/B comparison of 2 takes (current)
- PanoramaMix = multi-take audition with spatial positioning (future)
- Different use cases, different UX, different implementation

**Implication:** Do not conflate compare with PanoramaMix. Compare remains simple A/B toggle.

### 8.5 X/Y as Pluggable Evidence

**Decision:** X/Y capture is a pluggable evidence type, not hardcoded into scenarios.

**Rationale:**
- X/Y is one of many evidence types (pitch, timing, self-assessment)
- Evidence collection is orthogonal to scenario structure
- Future evidence types should be easy to add

**Implication:** Scenarios declare evidence requests, evidence collectors are pluggable.


### 8.6 Guided Authoring Before Graph Editor

**Decision:** Teacher authoring starts with guided forms, not graph editor.

**Rationale:**
- Graph editors are complex, intimidating for non-technical teachers
- Guided forms with semantic dimensions are more accessible
- Graph editor can be added later for advanced users

**Implication:** First authoring UX is form-based (dropdowns, sliders, toggles), not node-graph.

### 8.7 Alternation Special Family

**Decision:** Alternation (call-response, phrase-echo) is a special family, not a dimension of other families.

**Rationale:**
- Alternation requires fundamentally different orchestration (in-flight capture, continuous transport)
- Cannot be expressed as parameter variation of Echo family
- Requires unique runtime contracts (no pause mid-window, round-capture state)

**Implication:** Alternation is its own generator family with dedicated logic.

### 8.8 Tempo Ladder Experimental

**Decision:** Tempo ladder (progressive tempo increase) is experimental, not stable.

**Rationale:**
- Requires tempo change during quest execution (not yet implemented)
- Unclear if learners want automatic tempo progression vs manual control
- Needs user research before stabilizing

**Implication:** Tempo ladder family is marked experimental, may be removed or redesigned.

### 8.9 Sharing Unit Later = Card/Spec, Not Runtime State

**Decision:** When sharing is implemented, share QuestCard or ScenarioSpec, not QuestRun or runtime state.

**Rationale:**
- Runtime state is ephemeral, not portable
- Card/spec is declarative, portable, versionable
- Sharing runtime state couples sender/receiver implementations

**Implication:** Sharing infrastructure serializes cards/specs, not runs/programs.

### 8.10 Default Learner-Facing Quest Surface Remains Minimal

**Decision:** Default quest launcher shows minimal stable set (2-3 quests), not full catalog.

**Rationale:**
- Overwhelming learners with 20+ quests is bad UX
- Most learners want "just practice this block" with minimal choice
- Advanced users can access full catalog via separate UI

**Implication:** RecipeCardPopover (or successor) shows curated stable set, full catalog is opt-in.

## 10. What NOT To Do

These are **anti-patterns** — explicitly rejected approaches that violate system principles.

### 10.1 Hardcoded Recipe Explosion

**Anti-pattern:** Creating a new hardcoded recipe for every practice variation.

**Why wrong:**
- Doesn't scale (leads to 50+ recipes)
- Duplicates orchestration logic
- Makes teacher authoring impossible
- Couples UI to implementation

**Correct approach:** Create generator families that produce variations from semantic specs.

### 10.2 Graph Editor First

**Anti-pattern:** Building node-graph editor as first teacher authoring UX.

**Why wrong:**
- Too complex for non-technical teachers
- Intimidating for beginners
- Requires understanding of orchestration internals
- High implementation cost, low adoption

**Correct approach:** Start with guided forms (dropdowns, sliders, toggles), add graph editor later for advanced users.

### 10.3 Mix Compare with PanoramaMix

**Anti-pattern:** Conflating take compare (A/B toggle) with PanoramaMix (multi-take audition).

**Why wrong:**
- Different use cases (compare = 2 takes, PanoramaMix = N takes)
- Different UX (compare = toggle, PanoramaMix = spatial positioning)
- Different implementation (compare = simple overlay, PanoramaMix = complex mix engine)
- Premature coupling

**Correct approach:** Keep compare simple (A/B toggle), implement PanoramaMix separately when needed.

### 10.4 Pretend Current V-Mix Solves Recorded-Take Audition

**Anti-pattern:** Claiming current vocal mix slider solves multi-take audition problem.

**Why wrong:**
- V-Mix controls reference vocal stem, not recorded takes
- Recorded takes need spatial positioning, not volume slider
- Multi-take audition is a separate feature (PanoramaMix)
- Conflates reference playback with take audition

**Correct approach:** Acknowledge V-Mix is for reference stems, PanoramaMix is future feature for take audition.


### 10.5 Store Runtime State in Card

**Anti-pattern:** Storing QuestRun state (current round, step, evidence) in QuestCard.

**Why wrong:**
- Card is shareable metadata, run is ephemeral state
- Couples card to runtime implementation
- Makes sharing impossible (runtime state is not portable)
- Violates separation of concerns

**Correct approach:** QuestCard is immutable metadata, QuestRun is mutable state, keep separate.

### 10.6 Embed Generation Logic in Card

**Anti-pattern:** Putting scenario generation logic inside QuestCard.

**Why wrong:**
- Card is data, generator is code
- Makes cards non-portable (code cannot be serialized)
- Couples card to implementation language
- Violates separation of concerns

**Correct approach:** QuestCard references ScenarioSpec, ScenarioGenerator generates program from spec.

### 10.7 Skip Validation

**Anti-pattern:** Generating scenario programs without validating specs.

**Why wrong:**
- Invalid specs produce broken programs
- Runtime errors are harder to debug than validation errors
- Teachers get confusing error messages
- Violates fail-fast principle

**Correct approach:** Generators validate specs before generating programs, return clear validation errors.

### 10.8 Couple Quest to Takes Implementation

**Anti-pattern:** Quest runtime directly manipulating Takes internals (slots, recorder, assets).

**Why wrong:**
- Violates layering (Quest → Scenario → Takes)
- Makes Takes refactoring break Quest
- Duplicates orchestration logic
- Tight coupling

**Correct approach:** Quest runtime executes Scenario programs, Scenario orchestrates Takes operations.

### 10.9 Automatic Progression Without Evidence

**Anti-pattern:** Implementing adaptive difficulty without evidence collection.

**Why wrong:**
- Progression without evidence is guessing
- Cannot measure mastery without data
- Learners don't trust "magic" progression
- Violates transparency principle

**Correct approach:** Collect evidence first, build progression model on evidence, make progression transparent.

### 10.10 Overwhelming Default Quest Surface

**Anti-pattern:** Showing 20+ quests in default launcher.

**Why wrong:**
- Paralyzes learners with choice
- Most learners want "just practice this block"
- Hides stable, proven quests in noise
- Bad UX for beginners

**Correct approach:** Default launcher shows 2-3 stable quests, full catalog is opt-in for advanced users.

## 13. Key Takeaways

### For Implementers

1. **Generators are the scaling unit** — not recipes, not hardcoded programs
2. **Specs are declarative** — semantic dimensions, not imperative steps
3. **Families are rare** — 3-7 families, not 20+, must have distinct orchestration logic
4. **Separation is strict** — Card ≠ Program ≠ Run ≠ Generator
5. **Takes is substrate** — Quest/Scenario orchestrates, doesn't replace
6. **Evidence is pluggable** — X/Y, pitch, timing, self-assessment are all evidence types
7. **Capability gating is first-class** — scenarios adapt to available features
8. **Interruption is contractual** — program defines policy, runtime enforces

### For Designers

1. **Start with guided forms** — not graph editor, not code
2. **Semantic dimensions are orthogonal** — scope, tempo, backing, etc. are independent
3. **Default surface is minimal** — 2-3 stable quests, full catalog is opt-in
4. **Progression is transparent** — learners see evidence, understand why system suggests next practice
5. **Quest is motivational** — name, icon, goal framing matter
6. **Scene policy is declarative** — scenarios control visibility, highlighting, cues

### For Teachers

1. **Compose from dimensions** — not code, not graph nodes
2. **Specs are portable** — share cards/specs, not runtime state
3. **Validation is immediate** — know if spec is valid before testing
4. **Preview before publish** — test scenarios before sharing
5. **Evidence is visible** — see what mastery signals are collected
6. **Progression is optional** — manual authoring works, adaptive is future

### For Learners

1. **Quests are practice sessions** — structured, goal-oriented, evidence-based
2. **Default is simple** — 2-3 stable quests, no overwhelming choice
3. **Progress is visible** — see rounds, steps, completion, evidence
4. **Interruption is safe** — Escape key works, state is preserved (future)
5. **Evidence is transparent** — see what system measures, why it suggests next practice (future)

## 14. Glossary

**Alternation Pattern** — Guide/user alternation structure (call-response, phrase-echo, etc.)

**Backing Mode** — Which audio stems are active (full, instrumental, vocals-only, none)

**Capability Gating** — Adapting scenarios to available features (word-sync, pitch, backing stems)

**Evidence** — Captured mastery signals (completion, X/Y, pitch, timing, self-assessment)

**Generator Family** — Family-specific logic for generating scenario programs from specs

**Hold/Fix Behavior** — Error handling strategy (pause-on-error, continue, retry)

**Listen Mode** — How learner hears reference before recording (reference, guide, none)

**Mastery** — Aggregated mastery state for a song/block (scores, progression state)

**Progress Gate** — Completion criteria (rounds, filled-slots, mastery-threshold, time-limit)

**Quest Card** — Motivational wrapper + shareable unit (name, icon, description, spec reference)

**Quest Run** — Runtime instance of a quest execution (program + state + evidence)

**Record Mode** — Recording capture strategy (standard, in-flight, continuous)

**Review Step** — Post-recording review behavior (compare, self-assess, none)

**Scenario Program** — Executable step sequence with runtime contracts

**Scenario Spec** — Declarative specification of a practice scenario

**Scene Policy** — Visual presentation control (standard, minimal, guided, quest-overlay)

**Scope** — What portion of song to practice (block, line-range, full-song, custom)

**Tempo Stage** — Playback speed for difficulty scaling (1.0, 0.85, 0.70, ladder)
