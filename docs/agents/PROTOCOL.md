## Архитектура оркестрации

```
Никита → задача
    ↓
007 (диспетчер + разведчик)
    ↓
Выбирает скаутов из КАТАЛОГА (фиксированный список)
    ↓
Скауты читают doc + grep code → отчёт в Матрицу
    ↓
007 собирает MACRO-PACK (код + Context7 + Event map + Frozen guard)
    ↓
Центр (GLM-5.1 / Kimi) принимает архитектурное решение → TC Roadmap
    ↓
007 → берёт TC-1 → MICRO-PACK
    ↓
Оператор (изолированный) → cold verify → apply → tsc → vitest → стоп
    ↓
007 → verify (git diff + tsc) → docs sync → AUDIT.md
    ↓
009 → runtime diagnosis (manual / automated)
    ↓
Quality Gate → git push
    ↓
Все TC ok → Центр закрывает задачу → Никита тестит
```

## Золотые правила

### 1. Скауты только читают

```
Скаут
    ↓
анализирует
    ↓
отдаёт отчёт

НЕ

    ↓
решает
    ↓
создаёт агента
    ↓
меняет код
```

**Нарушение = немедленная блокировка скаута.**

### 2. Фазовое внедрение

| Фаза | Скауты | Когда запускать |
|------|--------|----------------|
| Phase 1 | arch-scout, sync-scout, gateway-scout | Прямо сейчас |
| Phase 2 | audio-scout, perf-scout | Когда Phase 1 стабильна 2 недели |
| Phase 3 | ui-scout, billy-scout | По запросу Центра |

**Запрещено запускать Phase N+1 до стабилизации Phase N.**

### 3. Каталог фиксирован

007 выбирает скаутов **только из заранее объявленного REGISTRY.md**.
Скаут не может создать субагента.
Скаут не может изменить свой документ.

### 4. Модельная дисциплина

| Задача | Модель | Почему |
|--------|--------|--------|
| Разведка, рутина | GLM-5.1 (T0) | Бесплатно, быстро, хороший кодинг |
| Длинный контекст (>10K строк) | Kimi K2.6 (T3) | 2M контекст, лучшее качество на длинных документах |
| Глубокий анализ performance | DeepSeek V3 (T1) | Дешёвый, сильный reasoning |
| AI-код, промпты | GPT-4o-mini (T2) | Хорошо понимает AI-логику |
| Критические решения | Claude 3.5 Sonnet (T4) | Только с OVERRIDE от Центра |

**Запрещено использовать T4 без явного разрешения Центра.**

### 5. Quality Gate (после КАЖДОГО TC)

```
Оператор применил TC
    ↓
007 проверяет:
    1. git diff — корректно?
    2. tsc --noEmit — 0 errors?
    3. vitest run --related — passed?
    4. Frozen guard — ничего не тронуто?
    5. AUDIT.md — записано?
    ↓
Всё ok → push
    ↓
009 проверяет runtime
    ↓
Баги → новый TC
```

### 6. Docs Sync (после push)

**Триггер: Никита сказал "запушь"**

```
1. git diff — что изменилось
2. Для каждого изменённого файла:
   - src/bridges/*.ts → bridge-*.md в BeLive_CEO/
   - src/stores/*.ts → store-заметка
   - src/services/*.ts → сервис-заметка
   - DB_VERSION? → frozen-decisions-index
   - Новый паттерн? → P-XX.md + PATTERN-REGISTRY
3. Записать в AUDIT.md
4. Обновить MATRIX.md (если doc-code соответствие изменилось)
```

## Форматы

### MACRO-PACK (007 → Центр)

```markdown
# PACK: task-xxx
## ЗАДАЧА: [от Никиты]
## АРХИТЕКТУРНЫЙ САММАРИ: [выдержки из доков]
## FROZEN GUARD: OK / PARTIAL / INTRUSION
## CONTEXT7: [альтернативы]
## КОД: [полные файлы]
## ГРАФ: [imports, events, stores]
## git diff: [если есть]
## STATE SNAPSHOT: [perf budget]
```

### MICRO-PACK (007 → Оператор)

```markdown
# MICRO-PACK: TC-XXX
ЦЕЛЬ: [одна фраза]
ФАЙЛ: path/to/file.ts
ЗОНА: lines N-M
ДЕЙСТВИЕ: [точный код]
КОНТЕКСТ: [3 строки до/после]
ЗАПРЕЩЕНО: [что не трогать]
ТЕСТ: tsc --noEmit && vitest run --related
```

### Отчёт скаута (Скаут → Матрицу)

```markdown
# SCOUT-REPORT: D[X] [домен]
ДАТА: YYYY-MM-DD
СКАУТ: [имя]
ДОКУМЕНТ: [путь]
СТАТУС: OK / MINOR / DRIFT / BROKEN / FROZEN
ПОКРЫТИЕ: X% (сколько из документа реализовано в коде)
РАСХОЖДЕНИЯ:
- [строка в доке] vs [строка в коде] → [объяснение]
ЧТО ДЕЛАТЬ:
- [конкретное действие или "ничего"]
ТОКЕНОВ ПОТРАЧЕНО: [N]
```

## Почему это работает

**Сильная сторона:**
- Не агенты ради агентов, а **система знаний**
- Матрица doc-code покажет дрифт ДО того как он сломает прод
- AUDIT.md даст историю через полгода
- Фазовое внедрение = никакого зоопарка

**Что избегаем:**
- "Агенты создают агентов" → хаос, потеря контроля
- 15 скаутов сразу → бюрократия, шум
- Сложный cost-dash → оверинжиниринг
- Автоматизация без измеримого выигрыша
