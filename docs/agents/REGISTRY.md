## Принцип работы

Каждый **домен** проекта beLive закреплён за **скаут-агентом**.
Скаут читает свой документ → сверяет с кодом → отправляет отчёт в **Матрицу**.

**Золотое правило:** Скауты только читают и отдают отчёт. **Не решают. Не создают агентов.**

```
Документ → Скаут (читает + grep) → Отчёт соответствия → Матрица → Центр
```

## Почему именно эти фазы?

### Phase 1: Максимум информации

- **arch-scout** → видит ВЕСЬ проект сверху. Дает макро-отчёт: "всего 352 файла, док говорит 170, что не учтено"
- **sync-scout** → самый критичный домен. Sync = сердце beLive. Если doc и code расходятся — проект ломается
- **gateway-scout** → инфраструктура: auth, AI gateway, безопасность. Без этого всё остальное не работает

### Phase 2: Глубокие домены

- **audio-scout** → AudioEngineV2 (❄️ frozen). Нужен когда будем менять audio-контракты
- **perf-scout** → Performance budgets, runtime метрики. Нужен при оптимизации

### Phase 3: Продуктовые

- **ui-scout** → Practice surface, темы. Второстепенно для текущей фазы
- **billy-scout** → AI-ассистент. Меняется часто, док может устареть быстро

## Модельная стратегия

### T0: GLM-5.1 (opencode, бесплатно)
Все скауты по умолчанию. Основная рабочая лошадка:
- Чтение doc + grep кода → отчёт
- Сравнение doc↔code → обнаружение расхождений
- Ежедневная разведка

### T1: DeepSeek V3 ($0.14-0.28/1M)
- perf-scout: глубокий анализ performance budgets
- Когда GLM не справляется с тонкими несоответствиями

### T2: GPT-4o-mini ($0.15-0.60/1M)
- billy-scout: анализ AI-кода, промптов, логики
- Быстрая валидация сгенерированного кода

### T3: Kimi K2.6 (контекст 2M)
- arch-scout: архитектурный документ + весь связанный код
- Документы > 10K строк

### T4: Claude 3.5 Sonnet ($3-15/1M) — ЦЕНТР ТОЛЬКО
- Финальные архитектурные решения
- Разрешение конфликтов между скаутами
- OVERRIDE frozen-зон

## Escalation Protocol

```
T0 скаут → нашёл проблему → докладывает 007
007 → формирует MACRO-PACK → передаёт Центру
Центр → если нужно глубокое → T1/T2 скаут (по запросу)
Центр → если критическое → T4 анализ
Центр → TC → 007 → MICRO-PACK → Оператор
009 → проверяет runtime после push
```

---

## Дисциплина

**Запрещено:**
- Скаут сам создаёт субагента
- Скаут принимает архитектурное решение
- Более 3 скаутов активно в Phase 1
- Использование T4 без OVERRIDE от Центра
- Запуск Phase 2 до стабилизации Phase 1

**Обязательно:**
- После каждого крупного изменения → запись в AUDIT.md
- После каждого push → docs sync
- Quality gate: tsc + vitest + manual
